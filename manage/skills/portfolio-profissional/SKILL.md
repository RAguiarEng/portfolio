## Metodologia Padrão para Todos os Projetos

> **Esta seção é a referência inicial para qualquer novo projeto.**
> Ao criar um novo projeto e acionar esta skill, aplicar obrigatoriamente:
> 1. Criar a pasta `history/` com o arquivo `changelog.md`
> 2. Criar o Kanban e o Backlog conforme estrutura abaixo
> 3. Definir o MVP antes de qualquer linha de código

### Estrutura Kanban Padrão
```
BACKLOG → A FAZER (semana) → EM ANDAMENTO → REVISÃO/TESTE → CONCLUÍDO
```

**Toda segunda-feira:**
1. Consultar o Backlog
2. Escolher de 3 a 5 tarefas mais importantes para a semana
3. Mover para "A Fazer da semana" — anotar brevemente *por que foram escolhidas*

**Toda sexta-feira (opcional, mas recomendado):**
1. Commitar o que foi concluído
2. Registrar no `history/changelog.md`
3. Devolver pendências ao Backlog com prioridade alta
4. Avaliar se o cronograma precisa de ajuste

### Definição de MVP (padrão)
> **MVP = versão mais simples do sistema que entrega valor real ao usuário final, pronta para teste.**
> Não inclui features avançadas, otimizações ou integrações complexas — essas entram na Fase 2.

### Convenção de Commits (Conventional Commits)
| Tipo | Quando usar |
|---|---|
| `feat` | Nova funcionalidade |
| `fix` | Correção de bug |
| `refactor` | Refatoração sem mudança de comportamento |
| `docs` | Documentação |
| `style` | Formatação, sem mudança de lógica |
| `chore` | Manutenção (migrations, dependências) |

Exemplo: `fix: corrige meta description ausente em index.html e quemsou.html`

### Estrutura de `history/changelog.md` (padrão)
```markdown
## AAAA-MM-DD

### Modificação
- **Arquivos:** lista dos arquivos alterados
- **Resumo:** descrição muito sucinta da atividade

---

## AAAA-MM-DD HH:MM — Commit

### Commit — nome-do-repositorio
- **Mensagem:** `mensagem exata do commit`
```

**Diagnóstico do post:**
O texto funcionou porque tem uma estrutura clássica de alto engajamento: citação provocativa de autoridade → fato concreto → lição universal → implicação para o leitor. A abertura com aspas diretas do CEO da GranBio é um gancho forte — cria tensão imediata.

**Por que só 3 visitas ao perfil apesar de 673 alcançados:**
Esse é o dado mais revelador. O post gerou curiosidade intelectual, não curiosidade sobre *você*. O leitor saiu pensando "que caso interessante" — não "quem é esse cara que está me contando isso e o que ele faz?". Isso acontece porque o post não conecta o conteúdo à sua identidade profissional. Não há nenhuma âncora do tipo "aprendi isso trabalhando 10 anos em laboratório de multinacional" ou "esse tipo de problema é exatamente o que ajudo empresas a evitar".

**O que ajustar nos próximos posts:**
- Inserir uma frase de ancoragem pessoal — conectar o caso ao que você viveu ou faz
- O CTA ao final precisa ser de perfil ("me conta o que você achou nos comentários" ou "se quiser entender como isso se aplica ao seu processo") — não só um link de fonte

### Posicionamento de Conteúdo — O que fazer

**Fórmula central de cada publicação:**
> **Situação real → Problema concreto → O que foi feito → Resultado mensurável → Implicação para o leitor**

O leitor precisa sair do post pensando: *"isso acontece comigo também — esse cara pode me ajudar."*

**Formatos preferidos (baseado nas suas preferências):**

**1. Post curto de resultado (antes/depois)**
Estrutura: 3–5 parágrafos. Sem jargão. Número concreto no título ou na primeira linha.
Exemplo de abertura que funciona:
> *"Um laboratório estava demorando 4 dias para responder ao cliente interno. Depois de 3 semanas de trabalho, esse tempo caiu para menos de 18 horas. Veja o que mudou."*

**2. Carrossel com dados e gráfico**
Estrutura: 6–8 slides. Slide 1 = resultado/gancho. Slides 2–6 = desenvolvimento com dado visual. Slide final = CTA.
Ideal para casos com números, comparações, evolução no tempo.
